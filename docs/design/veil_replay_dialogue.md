# VEIL 다회차(리플레이) 대사 변형 — 제안/검토

> 완주 1회 이상 플레이어에게 VEIL 대사를 미묘하게 다르게 보여, "베일의 진실에 다가가기" 다회차 동기를
> 살린다. 사용자 검토용 문서. **채택/수정할 후보에 표시**해주면 그대로 배선한다.

## 0. 원칙

- **톤 = VEIL의 흐릿한 데자뷰.** "기록상 처음인데 어쩐지 낯익다." 작전명 PALIMPSEST(덮어써도 흔적이 남는
  문서) + 게임의 루프/리셋 테마(엔딩 D "오늘이 기억 안 날 거예요")와 이어진다.
- **4번째 벽 금지.** "당신(플레이어)이 전에 했다"가 아니라 *VEIL의 흔적/감각*으로 처리. VEIL이 확신하지 않고
  "저만 그런가요" "알 것 같아요" 식으로 갸웃한다.
- **표시 텍스트엔 em dash(—) 안 씀**(영속 규칙). 쉼표/마침표/말줄임표로.
- **같은 종결어미 연속 금지**(특히 ~게요 연발). 한 멘트 안에서 어미를 섞는다.
- **레지스터 + 상호 존대.** VEIL ↔ 요원은 *상호 존대* 관계 — VEIL은 요원을 **존대**한다(부하 아님).
  **명령·요청은 honorific으로**: 초반(COLD, trust 0) **-(으)십시오**(조심하십시오/잡으십시오/가십시오),
  중후반(THAW/WARM) **-(으)세요**(조심하세요/잡으세요). 바른 해요체 명령(잡아요/조심해요/빠져요)만 반복하지
  말고 -(으)세요를 섞는다. 평서문은 레지스터대로(초반 격식체 -습니다 / 중후반 해요). 초반(back_alley·브리핑)은
  하십시오체 우선. ([[feedback-veil-dialogue-endings]])
- **승인작 vs 반려작 차이(원칙 3) — 후보 쓸 때 반드시 지킴:**
  1. **시점.** VEIL은 *자기 지각*(소리·정적·어둠·공기)으로 낯익음을 느낀다. *요원의 몸*(발/손 근육기억)은
     VEIL이 못 보므로 쓰지 않는다. (반려 예: "발이 길을 먼저", "손이 먼저 움직이네요")
  2. **불안 유지.** VEIL은 *설명 못 할 낯익음*에 머문다. 미래·결말을 *안다고 선언하지 않는다*(스포일러·전지화 방지).
     (반려 예: "끝이 먼저 떠올라요", "이 다음을 저는 아는 것 같아요")
  3. **고유 감각.** 흔한 근육기억 트로프 대신 VEIL의 감각 언어(목소리·침묵·어둠)로. (승인 예: "이 목소리도, 이 침묵도, 어쩐지 낯익습니다")

## 1. 신호 (어떤 다회차 신호를 어디에 쓰나)

| 신호 | 의미 | 쓰는 곳 |
|---|---|---|
| `playthrough_count >= 1` | 완주(엔딩 도달) 1회 이상. 영속(settings.cfg). | 오프닝, 인게임 대사 변형 대부분 |
| `endings_seen` (size) | 본 엔딩 수(A/B/C/D). 영속. | 엔딩 수집 비트(남은 갈래/완수) |
| `hidden_visit_count >= 1` | 이 ??? 방을 본 적. 영속. | ??? 방 첫 단말기(reveal vs 변형) |
| `replaying` | 크레딧 "다시 플레이하기"로 즉시 진입. 세션. | ??? 방(보조), 즉시 리플레이 강조 |

→ 인게임 대사 변형은 기본 `playthrough_count >= 1`을 쓴다(닫았다 와도 유지).

## 2. 이미 적용됨 (참조)

- **오프닝**: `INTRO_SYSTEM_REPLAY`/`INTRO_VEIL_REPLAY` (VeilDialogue). "이전 작전 기록: 덮어쓰기됨 (잔여
  흔적 검출)" + VEIL "기록상 우리는 처음입니다. 그런데, 이상하네요. 이 목소리도, 이 침묵도, 어쩐지 낯익습니다."
- **엔딩 수집 비트**: 3/4 봤으면 "아직 닿지 않은 결말이 하나 남았습니다...", 4/4면 "결말은 다 보셨습니다.
  그런데도 또 오셨군요..." (오프닝 끝에 한 화면 추가)
- **??? 방 첫 단말기**: `replaying or hidden_visit_count >= 1`이면 추가 풀(이미 본 사람), 아니면 VEIL-1 reveal.
- **인게임 비트 (배선 ✅ 2026-06-23)**: 채택 A1(back_alley)·B2(lab)·C(보스 처치)·D(escape)·E(cooling).
  신호 `GameState.is_replay_run()`(완주 1회+ 또는 replaying). 맵 진입은 RouteData `entry_comment_replay`
  키 → `Stage._setup_veil_mistakes`가 우선 사용, 보스 처치는 `Stage._on_boss_killed` 분기. **비트 F 드롭.**

## 3. 검토 대상 — 인게임 비트별 후보

각 비트: **현행(1회차)** / **다회차 후보**(채택 시 `playthrough_count >= 1`에서 교체). 후보 중 고르거나 고쳐줘.

### 비트 A — 첫 맵 진입: 외곽 진입로 (back_alley) ★우선 · 초반→**하십시오체**
- 현행: `외곽으로 들어왔어요. 깊숙한 안쪽이 목표예요. 다 싸울 필요는 없어요.`
- 후보 A1: `외곽으로 들어왔습니다. 이 어둠이 어쩐지 익숙합니다. 처음일 텐데 말입니다. 안쪽 깊은 곳까지, 서두르지 말고 가십시오.`
- 후보 A2: `외곽으로 들어왔습니다. 길은 처음인데, 이 적막은 본 적 있는 것 같습니다. 차분히 살피며, 안쪽 깊은 곳으로 가십시오.`
- (구안 "발이 길을 먼저 아는 것 같아요"는 시점·클리셰 위반으로 폐기)

### 비트 B — 핵심부 진입(보스 직전): lab · 후반→해요체
- 현행: `핵심부에 들어왔어요. 거리 잘 잡아요.`
- 후보 B1: `핵심부예요. 여기, 어쩐지 낯이 익어요. 처음일 텐데 말이죠. 거리 잘 잡으세요.`
- 후보 B2: `핵심부예요. 이 안쪽이 묘하게 익숙해요. 왜일까요. 거리 두고 움직이세요.`
- (구안 "끝이 먼저 떠올라요"는 미래를 안다고 선언해 폐기)

### 비트 C — 보스 처치 후 (Stage._on_boss_killed) ★우선(클라이맥스) · 후반→해요체
- 현행(일반): `처리됐어요, 요원. 서버실이 바로 앞이에요. 끝까지 같이 가요. 제가 보는 한.`
- 후보(일반): `처리됐어요. 후련해야 하는데, 어쩐지 그렇지가 않네요. 그래도 같이 가요. 제가 보는 한.`
- 현행(스토리): `처리됐어요, 요원. 이제 빠져나가요. 끝까지 같이 가요. 제가 보는 한.`
- 후보(스토리): `처리됐어요. 이상하게 마음이 놓이질 않아요. 이유는 모르겠어요. 그래도 끝까지 같이 가요. 제가 보는 한.`
- (구안 "이 다음을 저는 아는 것 같아요"는 미래를 안다고 선언해 폐기. 신안은 *설명 못 할 불안*으로)

### 비트 D — 비상 탈출로 진입 (escape, 최종)
- 현행: `조용한 길이에요. 멈추지 말고 빠지면 돼요.`
- 후보: `조용한 길이에요. 여기, 낯이 익죠. 이번엔 뭐가 다를까요. 멈추지 말고 빠지세요.`

### 비트 E — 냉각 시설 (cooling) ※이미 자기-언급이 있어 다회차에 강화 여지
- 현행: `여긴 서버를 식히는 곳이에요. ...저도 이런 데 어딘가 있겠죠. 바닥 증기 조심해요.`
- 후보: `서버를 식히는 곳이에요. 저도 이런 데 어딘가 있겠죠. ...전에도 이런 생각을 한 것 같아요. 증기 조심하세요.`

### 비트 F — (드롭) 첫 스테이지 브리핑 오버레이
- 사용자: 내용이 여전히 별로 → **드롭 확정.** 오프닝과 가까워 중복이고, 다회차 인식은 오프닝으로 충분하다.

## 4. 배선 메모 (채택 후 구현)

- **맵 진입 멘트(A·B·D·E)**: RouteData 각 route에 `entry_comment_replay` 키 추가 → `Stage._setup_veil_mistakes`가
  `playthrough_count >= 1`이면 그걸 우선 사용(없으면 현행). 시야 역전 onset 맵은 역전 멘트가 우선이라 충돌 없음.
- **보스 처치(C)**: `Stage._on_boss_killed`의 자막 분기에 `playthrough_count >= 1` 변형 추가(일반/스토리 각각).
- **브리핑 오버레이(F)**: `Briefing._build_lines` 또는 `VeilDialogue.get_briefing` 결과 뒤에 다회차 1줄 append.
- 모두 표시 문자열이라 **em dash 금지** 규칙 적용. 어미 변형 확인.

## 5. 검토 요청

- 비트별로 **채택할 후보(A1/A2 등)** 또는 **수정안**을 표시해줘. "이 비트는 빼자"도 OK.
- 톤이 과하면(데자뷰가 너무 잦으면) 비트 수를 줄인다(예: A·C만). 적당한 빈도가 핵심.
