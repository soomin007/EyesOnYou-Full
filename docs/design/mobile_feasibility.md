# 모바일(터치) 지원 가능성 조사

> 사용자 요청(2026-06-26): "다시 밖이라 플레이테스트 못 해. 이 게임 아직 모바일 지원 안 하잖아."
> → 외출 중에도 폰으로 플레이테스트할 수 있게 모바일을 붙일 수 있는지 *먼저 조사*. 본격 구현 전 판단 자료.
> **결론 먼저: 생각보다 가능성 높음(중간 난이도). 최대 난관이라 본 '조준'이 아예 없음.** 단 실기기 검증 필요.

## 1. 판정 — 가능, 중간 난이도

가장 큰 걱정이던 **"마우스 조준 슈팅을 터치로 어떻게?"가 무효** — 이 게임은 마우스 조준이 아니다.
`Player._fire`는 `b.dir = facing`(바라보는 방향 ±1)으로 **좌/우 횡사격하는 런앤건**이다(Player.gd). 조준 스틱·
조준 재설계가 통째로 불필요 → 터치 레이아웃이 *표준 모바일 플랫포머*(이동/점프/사격/대시/스킬 버튼)로 끝난다.

## 2. 이미 유리한 것 (추가 작업 0)

| 항목 | 현황 | 의미 |
|---|---|---|
| 렌더러 | `gl_compatibility` | 모바일 WebGL 호환 렌더러(데스크톱 전용 Forward+ 아님). 모바일 적합. |
| 웹 빌드 | github.io 배포·영속 검증됨 | 네이티브 앱 빌드 불필요 — **모바일 브라우저로 기존 웹 빌드 접속**이 경로. |
| 스레드 | `thread_support=false`(단일 스레드) | SharedArrayBuffer/COOP·COEP 강제 의존이 약해 모바일 브라우저 호환 ↑(iOS Safari 유리). |
| 스트레치 | `canvas_items` / `aspect=expand` | 해상도 가변 대응. 1280×720 설계가 폰 가로화면으로 스케일. |
| 마우스↔터치 | `emulate_mouse_from_touch` 기본 true | **메뉴·카드·버튼 탭이 자동으로 클릭 처리** — Title/RouteMap/LevelUp/Settings 등 UI는 탭으로 바로 동작(추가 작업 없음). |
| 아트 | 100% 절차적 코드 | 모바일용 별도 에셋 0. |

## 3. 새로 만들어야 하는 것 (그린필드 — 현재 터치 처리 코드 0)

인게임 조작만 터치 버튼이 필요하다(메뉴는 §2의 emulate로 해결). 필요한 입력 액션 = 9개 중 인게임:
`move_left / move_right / jump / attack(꾹=연사) / dash / skill / move_down(하향 통과) / pause`.

- **온스크린 터치 컨트롤 오버레이**: 좌측 = 좌/우 이동(또는 가상 디패드), 우측 = 점프·사격·대시·스킬 클러스터,
  상단 구석 = 일시정지(ESC 대체). `TouchScreenButton` 또는 액션을 쏘는 Control 버튼. 인게임 HUD(Stage)에 얹음.
- **표시 게이팅**: `DisplayServer.is_touchscreen_available()`(또는 설정 토글)로 터치 기기에서만 노출 — 데스크톱
  플레이 무영향.
- **브리핑/자막 진행**: 현재 점프/스페이스/ui_skip로 넘김 → 터치는 화면 탭으로 진행 1줄 추가(또는 버튼).
- **가로 잠금**: PWA `orientation`/HTML로 landscape 고정(세로면 조작 불가).
- **HUD 가독성**: 작은 화면에서 체력/스킬 HUD와 터치 버튼이 플레이필드를 가리지 않게 배치 조정.

## 4. 리스크 / 미지수 (실기기로만 확인 가능)

- **모바일 웹 터치 신뢰성**: 커뮤니티 보고상 모바일 브라우저 웹 export의 일부 터치/제스처 인식이 불안정할 수
  있음(특히 iOS Safari). 단순 탭/버튼은 대체로 OK지만 **실기기 검증 필수.**
- **iOS Safari**: 오디오 autoplay 정책(백로그 기존 웹 점검 항목), 풀스크린 제약, 주소창 뷰포트 변동, 100vh 이슈.
- **성능**: 저사양 폰 브라우저 프레임. 절차적 2D라 가벼운 편이나 미검증.
- **정밀 조작 손맛(설계 리스크)**: coyote-time·대시·글라이드가 있는 *정밀* 런앤건이라, 터치 디패드는 키보드보다
  정밀도가 낮아 난이도 체감이 나빠질 수 있음. 이건 엔지니어링이 아니라 *플레이 느낌* 문제 → 스파이크로만 판정.

## 5. 권고 — 최소 스파이크로 결정

본격 구현 전, **테스트 가능한 러프 빌드**를 빠르게 만들어 사용자가 폰으로 직접 확인하는 게 가장 값싼 결정법:

1. 기본 터치 오버레이(이동/점프/사격/대시/스킬/일시정지) + 터치 기기 게이팅.
2. 웹 export → github.io 배포.
3. **사용자 폰 실측**: ① 입력이 먹나(iOS/안드로이드) ② 정밀 조작 손맛 ③ 성능 ④ 버튼이 시야 안 가리나.

→ 스파이크가 "되네 + 할 만하네"면 폴리시 투자, "터치로는 손맛이 죽네"면 재고. **러프 빌드 ≈ 반나절~하루,
폴리시(레이아웃·햅틱·정밀 튜닝)는 별도 다회 세션.** 이게 사용자의 "외출 중 플레이테스트" 블로커를 영구
해소하는 가장 직접적 길이며, 동시에 데모/본편의 *플레이어 저변*(폰 접근성)도 넓힌다.
